# Trace ID Conventions

llmtrace follows OpenTelemetry conventions for trace correlation:

- **trace_id** — Shared by all events in the same trace. Generated by the root span, inherited by all descendants.
- **span_id** — Unique identifier for each event/span. Every TraceEvent has its own span_id.
- **parent_id** — References the parent span's span_id. Null for root spans.

## Example Trace Tree

```
@trace("agent")              trace_id=T, span_id=A, parent_id=null
  ├── LLM call               trace_id=T, span_id=B, parent_id=A
  ├── tool: get_weather       trace_id=T, span_id=C, parent_id=A
  └── LLM call               trace_id=T, span_id=D, parent_id=A
```

## How It Works

### Root Span
When `@trace` is called with no existing trace context, it creates a new `trace_id` and sets `parent_id=None`. This is the root of the trace tree.

### Child Spans
When `@trace`, `@trace_tool`, or an instrumentor runs inside an existing trace context:
- `trace_id` is inherited from the parent (shared across the entire trace)
- `span_id` is a new unique UUID for this specific event
- `parent_id` is set to the parent's `span_id`

### Context Propagation
Trace context is propagated via Python `contextvars`:
- `_current_trace_id` — the shared trace ID for the current trace
- `_current_parent_id` — the current span's ID (becomes `parent_id` for children)
- `_current_span_id` — the current span's ID (for `span()` context manager compat)

### OTLP Export
The OTLP sink maps these IDs directly to OpenTelemetry's trace model:
- `event.trace_id` → OTel `trace_id` (128-bit)
- `event.span_id` → OTel `span_id` (64-bit, lower bits of UUID)
- `event.parent_id` → OTel parent span context
